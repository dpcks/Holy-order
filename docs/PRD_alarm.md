Holy-Order 사용자 페이지(`frontend/src/pages/OrderStatus.tsx`)의
주문 완료 알림(소리/진동/시각) 동작을 iPhone 환경까지 고려해서 개선해줘.

현재 문제점:
- iPhone Safari는 사용자 상호작용 전에 오디오 재생이 차단됨
- iPhone Safari는 navigator.vibrate를 지원하지 않음
- iPhone Safari는 일반 Notification API를 지원하지 않음
- 현재 알림 재생은 사용자 상호작용 없이 호출되어 iOS에서 대부분 차단됨

이번 요청은 새 기능을 만드는 게 아니라
"현재 알림 로직을 iOS + Android 양쪽에서 최대한 잘 동작하게 개선하는 작업"이야.

기존 기능, 구조, UI, 라우팅, WebSocket, 폴링 로직은 최대한 유지해줘.
불필요한 리팩토링은 하지 말고 요청한 부분만 정확히 개선해줘.

--------------------------------------------------

[요구사항]

1. 오디오 언락 로직 추가 (가장 중요)
- 사용자가 페이지에서 처음 터치/클릭할 때
  무음 재생 → 즉시 pause 방식으로 오디오를 미리 언락해둘 것
- iOS Safari에서 이후 알림음이 정상 재생될 수 있도록 할 것
- 이벤트 리스너는 click, touchstart 둘 다 등록
- 언락 성공 후 리스너는 제거
- 컴포넌트 언마운트 시 리스너 정리 반드시 포함

2. 알림 재생 시 재시도 로직 추가
- 주문 상태가 READY로 바뀌면 알림음 재생
- 재생 실패 시 200ms 후 1회 재시도
- 그래도 실패하면 console.warn으로만 기록
- 알림음은 currentTime = 0으로 초기화 후 재생

3. 진동 처리
- navigator.vibrate가 존재하는 경우에만 진동
- iOS는 진동 불가이므로 조건부로 무시
- 진동 대신 강한 시각 피드백을 제공할 것

4. iOS 대체 시각 피드백 추가
- READY 상태로 전이되는 순간 다음을 실행:
  - 화면 상단/중앙에 큰 알림 오버레이 또는 배너 표시
  - 배경 또는 카드 요소를 2~3초 강한 펄스/깜빡임 애니메이션
  - 필요 시 화면 전체에 짧은 플래시 효과(흰색/엑센트 색) 한 번
- iOS에서도 알림 효과가 분명하게 느껴지도록 구성
- 기존 주문 상태 UI와 자연스럽게 어울려야 함
- 과도한 모달은 지양

5. Notification API 호환성 정리
- 'Notification' in window 체크
- permission이 granted인 경우에만 사용
- iOS Safari처럼 지원 안 되는 환경에서는 무시
- try/catch로 예외 막기

6. 알림 관련 useEffect 정리
- prevStatusRef 로직 유지
- READY 진입 시 단 1회만 알림
- 이미 READY였던 상태에서 재렌더로 인해 중복 알림 발생하지 않도록 처리
- 상태가 CANCELLED, COMPLETED 등으로 빠질 때 ref 초기화 처리 검토

7. 오디오 파일 경로 안정화
- 현재 외부 URL을 사용하는데,
  `/mp3/ready.mp3` 같은 로컬 경로로 전환 가능한 구조로 정리할 것
- 로컬 파일이 없다면 기존 URL을 유지하고 주석에 TODO를 남길 것
- 공통 유틸로 분리 가능하면 `utils/notify.ts` 정도로 추출해도 됨

8. iOS 안내 UX 추가
- 페이지 진입 시 iPhone 사용자에게
  “알림음을 받으려면 화면을 한 번 터치해주세요”
  수준의 짧은 안내 배너를 2~3초간 노출
- iOS 감지는 navigator.userAgent 기반 간단 판별 OK
- 이미 언락된 경우에는 배너 표시하지 않음
- UI 스타일은 기존 톤과 일치시킬 것
