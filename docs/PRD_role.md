# [PRD] 관리자 권한 관리 시스템 (RBAC) 도입

## 1. 개요
- **목적**: 시스템 설정 및 계정 관리에 대한 접근 권한을 차등 부여하여 보안을 강화하고 운영 실수를 방지한다.
- **대상**: Holy-Order 시스템 관리자 페이지를 사용하는 모든 관리자.

## 2. 권한 정의
권한은 크게 두 단계(`MASTER`, `ADMIN`)로 구분한다.

### 2.1 MASTER (최고 관리자)
- **정의**: 시스템의 모든 설정을 변경하고 계정을 관리할 수 있는 계정.
- **주요 권한**:
  - 관리자 계정 생성 및 목록 조회.
  - 관리자 계정 활성화/비활성화 상태 변경.
  - 시스템 전역 설정 (영업 상태, 계좌 정보, 전화번호 필수 여부 등) 변경.
  - 모든 운영 데이터(주문, 매출, 재고 등) 접근 가능.

### 2.2 ADMIN (일반 운영자)
- **정의**: 매장 운영 및 주문 처리에 집중하는 현장 봉사자 계정.
- **주요 권한**:
  - 주문 관리 및 상태 변경 (준비 중, 완료 등).
  - 재고 관리 및 메뉴 가용성 설정.
  - 매출 통계 조회.
- **제한 사항**:
  - 타 관리자 계정 생성 및 상태 변경 불가능.
  - 주요 시스템 설정(계좌번호 등) 변경 제한 (조회만 가능) .

## 3. 기능 요구사항

### 3.1 백엔드 (FastAPI)
- **데이터베이스 모델**: `Admin` 테이블에 `role` 컬럼 추가 (Default: 'ADMIN').
- **스키마 업데이트**: `AdminResponse`, `AdminCreate` 등에 `role` 필드 반영.
- **인가(Authorization) 로직**:
  - `MASTER` 전용 API 엔드포인트 보호 (계정 생성/수정/시스템 설정 변경).
  - 현재 접속한 사용자의 `role`을 검증하는 의존성(`get_master_user`) 구현.

### 3.2 프론트엔드 (React)
- **계정 생성 UI**: 관리자 생성 시 `role`을 선택할 수 있는 Dropdown 메뉴 추가.
- **권한별 접근 제어**:
  - `ADMIN` 권한으로 로그인 시 '관리자 계정 추가' 버튼/폼 숨김 처리.
  - '시스템 설정' 내 민감한 데이터 수정 버튼 비활성화.
  - 사이드바나 프로필 영역에 현재 권한(`MASTER`/`ADMIN`) 표시.

## 4. 데이터베이스 마이그레이션 계획
- 기존의 모든 계정은 하위 호환성을 위해 초기 마이그레이션 시 `MASTER` 권한을 부여한다.
- 이후 생성되는 봉사자 계정은 목적에 맞게 `ADMIN` 권한을 할당한다.

## 5. 성공 지표
- 권한이 없는 사용자가 계정 추가 API를 호출했을 때 `403 Forbidden` 에러가 정상적으로 발생하는가.
- 권한에 따라 프론트엔드 UI가 동적으로 변화하여 사용자에게 혼선을 주지 않는가.
- 최고 관리자가 실수로 본인의 권한을 박탈하지 못하도록 보호 로직이 작동하는가.
